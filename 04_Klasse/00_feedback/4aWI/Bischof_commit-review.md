Four concrete strategies: `RightMovement`, `LeftMovement`, `UpMovement`, `DownMovement`. `Circle.setMovement()` allows runtime strategy swapping. `MyGame.init()` creates 4 circles each with a different movement strategy.
- Issues/Risks: `SuperCircle.update()` calls `super.update()` which just delegates to `movement.update()` — the override is a no-op and could be removed. `Circle` constructor default-constructs a `RightMovement` even when immediately overridden by `setMovement()` — wasteful. Each movement class stores its own `x, y` but the actor also has `x, y` fields — position is tracked in two places (in actor's constructor, in movement), which can diverge. `render()` uses `movement.getX/Y()` but `Circle.x/y` are never updated.
- Suggestions: Have the `Actor` delegate position entirely to the movement strategy, or have the strategy update the actor's position directly. Add boundary checks.
